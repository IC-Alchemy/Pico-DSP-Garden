# Pico-DSP-Garden — Layout, Architecture & Organization Review

## Summary
The DSP library (`rpdsp`) and audio driver are clean and well-namespaced, but the **repository wrapping them has accumulated real structural debt**: ~half the tracked files are hand-synced duplicates, build artifacts and machine-specific configs are committed, the `.gitignore` is broken, and the design docs describe an API that **does not exist in the code**. Below, grouped by theme and priority.

---

## 1. The canonical-copy duplication (biggest architectural issue)

**Evidence:** `audio/` + `dsp/` = 25 canonical files. Each of the 4 examples carries a full copy under `<example>/src/`. Currently in sync (verified byte-for-byte), but that's 4 × 25 = **~100 duplicate tracked files out of 191 total** — over half the repo is copies. AGENTS.md codifies this as policy, which makes the hazard *mandatory* rather than accidental.

**Why it's a problem:** The AGENTS rule "you must propagate the change to every example's `src/` copy or the examples will silently diverge" is exactly the kind of manual invariant that *will* break. The only thing preventing drift today is that the project has 25 commits. The `SuperSaw.h` example-local file already proves the model leaks (it lives only in one example's copy).

**Recommendation — convert `rpdsp` + `audio` into a real Arduino library:**
```
libraries/
  rpdsp/            <- library.properties + src/ (the canonical dsp/ tree)
  pico_audio_i2s/   <- src/ (the canonical audio/ tree)
```
- Add `library.properties` so the Arduino IDE / `arduino-cli` resolves them automatically.
- Examples then `#include <rpdsp/oscillator.h>` with **zero copies** — a single source of truth.
- Arduino libraries support per-example code that links a shared library natively; this is the idiomatic fix and eliminates the entire "propagate to every copy" rule from AGENTS.md.
- `SuperSaw.h` becomes a first-class `rpdsp::Hypersaw` in the canonical tree (promote it, per your own "if you promote it, add it to canonical `dsp/`" note).

If you want to keep examples self-contained for offline use, at minimum add a **sync script** (`scripts/sync-copies.ps1`) that copies canonical → examples and fails CI on drift, instead of relying on humans.

## 2. Committed build artifacts & machine-specific files

**Evidence (from `git ls-files`):**
- `build/` is **tracked** (17 files, ~550 KB) including `SuperSaw.ino.elf`, `.bin`, `.map`, plus *compiled copies of the duplicated sources* (`build/sketch/src/audio/...`). Meanwhile `.gitignore` contains `build/*` three times and `build\*` once — gitignore was added *after* committing, so the ignores do nothing for already-tracked files.
- `.vscode/` is tracked but ignored — and `c_cpp_properties.json` contains **absolute paths to `C:\Users\Blasto\...`** and `arduino.json` hardcodes `COM6` + `Examples\SuperSaw`. This will break for every other contributor.
- `.qoder/` is tracked (13 files of another tool's generated knowledge) but ignored.

**Fix:**
```powershell
git rm -r --cached build .vscode .qoder
```
Then clean up `.gitignore` (see §3). Keep `.vscode/arduino.json` and `c_cpp_properties.json` as `.example` templates, or use `${env:USERPROFILE}` / variable expansion.

## 4. Docs describe an API that does not exist

This is the most insidious issue — your **non-ignored** docs (`realtime_rules.md`, `algorithm_catalog.md`, `rp2350_tuning.md`) reference types and files that **do not exist in `dsp/`**:

| Documented (as `rpdsp::…`) | Actually in `dsp/`? |
|---|---|
| `DefaultAudioBlock`, `AudioBlock<Capacity>` | ❌ no block container type exists |
| `MidiByteParser`, `MidiNoteEvent`, `MidiControlEvent` | ❌ none |
| `SmoothedAdcParameter`, `ParameterCurve` | ❌ none |
| `GateDebouncer` | ❌ none |
| `CallbackPerformanceMeter` | ❌ none |
| `PioI2sAudio` (rp2350_tuning.md L84) | ❌ not in `audio/` |
| `examples/oscillator_gallery.cpp`, etc. | ❌ path wrong — they're in `Docs/example_Ideas/`, which AGENTS.md says to *ignore* |

Also the **baseline parameters in the docs contradict every example**:
- Docs: "approved baseline is 48 kHz, 32-sample blocks" (algorithm_catalog L3, rp2350_tuning L3, realtime_rules L3).
- Reality: 3 of 4 examples run **44100 Hz**; all use **256-sample buffers**; `RPDSP_BLOCK_SIZE` (32) is declared but **never used** by any example (they process per-sample with `N = max_sample_count = 256`).
- Docs use a `process(block)` processor-framework signature; examples use `fill_audio_buffer(buffer)`. No block-processing framework exists.

**Fix options:**
- (a) Implement the documented block framework + missing helpers, **or**
- (b) Rewrite the docs to describe what actually ships (per-sample `fill_audio_buffer`, `LinearSmoother`, etc.) and move the aspirational design to a clearly-labeled `Docs/roadmap.md`.

Either way, the current state actively misleads agents and contributors (AGENTS.md itself says "agents familiar with DaisySP will guess wrong" — your own docs now cause the same problem).

## 5. Missing LICENSE file

**Evidence:** No `LICENSE` at repo root. But AGENTS.md ("License: MIT"), README ("open-source... ready to fork"), and `rp2350_tuning.md` L12 all assert MIT. The README even links to a public GitHub repo. **A repo claiming to be open-source with no LICENSE file is, by default, all-rights-reserved.** This is a legal blocker, not a style issue.

**Fix:** Add a standard `LICENSE` (MIT) and add `SPDX-License-Identifier: MIT` headers to the canonical `dsp/` headers (the vendored `audio/` already carries BSD-3 notices — preserve those).

## 6. Duplicated example boilerplate (separate from §1)

Even setting aside the `src/` copies, every `.ino` reimplements the same scaffolding four different ways, and they've already drifted stylistically:
- `float_to_int16` exists in 3 examples under 3 names (`float_to_int16`, `convertSampleToInt16`, `toInt16`) with subtly different behavior (`roundf` in 2, not in the 3rd).
- I2S setup helper duplicated 4× — `Oscillators.ino` has leftover debug spam (`"We are melting!!!!!"`, `delay(1000)` inside an RT setup path), while `SuperSaw.ino`/`SimpleOscillators.ino` have a clean version.
- Pin constants declared `int` in two examples, `static const int` in the other two.

**Fix:** This scaffolding (I2S init, `float_to_int16`, the `producer_pool` loop) belongs in the shared `audio` library as a tiny `PicoDspGarden` helper — then each example shrinks to ~30 lines of *just its DSP*.

---

## 7. Flat `dsp/` layout has no internal organization

**Evidence:** 16 headers in one flat directory spanning oscillators, filters, dynamics, effects, envelopes, sequencing utilities, and UI helpers (`knob_bank.h`, `pickup_knob.h`, `joystick_recorder.h`). Once promoted to a real library (§1), group them:

```
rpdsp/src/rpdsp/
  synthesis/   oscillator.h, voice.h, noise...
  filters/     filter.h
  dynamics/    dynamics.h
  effects/     effects.h, delay_line.h
  modulators/  envelope.h, parameter_smoother.h
  sequencing/  rhythm_sequencer.h, gate_pattern.h
  ui/          knob_bank.h, pickup_knob.h, joystick_recorder.h
  core/        config.h, algorithm.h, realtime.h
```

This also gives you room for the missing block framework from §4 (`core/audio_block.h`).

## 8. Project is hardcoded to SuperSaw as "the" sketch

`.vscode/arduino.json`, `build/build.options.json`, and the committed `build/` all point at `Examples/SuperSaw/`. There's no way to build another example without editing tracked files. Once §2 removes the committed `build/`, the `.vscode` example should be a template, and each example directory should be independently openable (Arduino convention already supports this).

## 9. Minor / lower-priority

- `Docs/example_Ideas/` (21 files) — AGENTS.md says "IGNORE" but the canonical `algorithm_catalog.md` *cites files that live there*. Either fold them into real `Examples/` or move them out of Docs into an `attic/` and stop referencing them from shipped docs.
- `Docs/dsp_paper_shortlist.md` — check whether it belongs in a shipped repo or a personal notes location.
- `compile_commands.json` exists and is committed — you could add a trivial **header-compile CI job** (compile each `dsp/*.h` with the arm-gcc from the toolchain in `-fsyntax-only` mode). AGENTS.md says "no host build" — true for firmware, but a syntax-only check of a header-only library is cheap and would have caught the doc drift.
- Sample-rate inconsistency (48000 vs 44100) is documented as intentional; consider at least making it a single `#define` per example with a comment so it's visibly a *choice*.

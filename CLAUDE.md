# CLAUDE.md

Guidance for Claude Code (and other AI assistants) working in this repository.

## What this is

Pico-DSP-Garden is a C++17 Arduino framework for real-time audio DSP on the
Raspberry Pi Pico 2 (RP2350, primary target) and Pico (RP2040). It pairs a
self-authored, header-only DSP library (`rpdsp`) with a vendored PIO/DMA I2S
driver (`pico_audio_i2s`) and a handful of `.ino` example sketches that drive
a PCM510x-class stereo DAC.

There is no host build system, no test framework, and no CI. Verification
means compiling a sketch for the real target with `arduino-cli` (see
"Build & validate" below).

## Build & validate

No host tests exist. The only validation available is compiling for the real
target:

```sh
arduino-cli compile \
  --fqbn "rp2040:rp2040:rpipico2:flash=4194304_0,freq=200,opt=Optimize3,usbstack=picosdk" \
  --library libraries/rpdsp \
  --library libraries/pico_audio_i2s \
  Examples/<Name>
```

Run this for every example touched by a change. A quick syntax-only sanity
check for `rpdsp` header edits alone (no Arduino core needed) is an aggregator
translation unit including the headers you changed and `-fsyntax-only`, but
this does **not** replace the firmware compile above — the real GCC target,
flags, and `Arduino.h` differ.

Board/core: Arduino IDE 2.x with the **earle-philhower arduino-pico** core,
board **Raspberry Pi Pico 2**. To install the two libraries in the Arduino
IDE, copy or symlink `libraries/rpdsp` and `libraries/pico_audio_i2s` into the
sketchbook `libraries/` directory; `arduino-cli` instead takes `--library`
flags pointing at the repo paths directly (no copy needed).

## Hardware baseline

| Signal | GPIO | Notes |
|--------|------|-------|
| DATA   | 15   | `PICO_AUDIO_I2S_DATA_PIN` |
| LRCK   | 16   | `PICO_AUDIO_I2S_CLOCK_PIN_BASE` |
| BCLK   | 17   | driver-assigned, always `clock_pin_base + 1` |

All current examples run at `SAMPLE_RATE = 48000.0f` (this matches
`rpdsp::kDefaultSampleRate`). `SAMPLES_PER_BUFFER` varies per example (32 in
the oscillator demos, 256 in SuperSaw/LadderFilter) — it is not tied to
`RPDSP_BLOCK_SIZE` from `config.h`; examples process per-sample inside the
fill callback, not in `RPDSP_BLOCK_SIZE`-sized blocks. Treat sample rate and
buffer size as per-sketch constants set near the top of each `.ino`, not as
framework-enforced values.

## Real architecture pattern (read the code, not just Docs/)

Every example follows the same dual-core, per-sample shape — this is the
pattern to follow for new sketches:

```cpp
// Core 0 — setup() / loop(): owns the real-time audio path, must never block
void setup() {
    // prepare() all DSP objects once, set up audio_format_t / audio_buffer_format_t,
    // audio_new_producer_pool(), audio_i2s_setup(), audio_i2s_connect(), audio_i2s_set_enabled(true)
}
void loop() {
    audio_buffer_t *buf = take_audio_buffer(producer_pool, true);
    if (buf) {
        fill_audio_buffer(buf);   // per-sample loop over buf->max_sample_count, writes int16 L/R
        give_audio_buffer(producer_pool, buf);
    }
}

// Core 1 — setup1() / loop1(): sequencing, control, Serial debug — blocking is fine here
void setup1() { Serial.begin(115200); /* ... */ }
void loop1() {
    // update DSP parameters via plain setters (e.g. osc.setFreq(...)); blocking delay() is OK
}
```

`fill_audio_buffer()` is the hot path: it converts DSP `float` output to
`int16_t` (clamp via `rpdsp::clamp(scaled, INT16_MIN_F, INT16_MAX_F)`) and
writes interleaved L/R samples directly into `buffer->buffer->bytes`. There is
no `AudioBlock`/`process(block)` framework in the actual code — see "Docs vs.
reality" below.

Cross-core state is shared through plain `volatile` variables or simple
setters on DSP objects (e.g. `volatile int note_index` written by Core 1, read
by Core 0). Keep cross-core messages small and avoid blocking Core 0 on
anything Core 1 owns.

## rpdsp library conventions

When adding or modifying a module in `libraries/rpdsp/src/rpdsp/`:

- **Namespace** everything under `rpdsp`. **Filenames are lowercase**
  snake_case (`ladder.h`, `gate_pattern.h`), even when the class name is
  CamelCase (`LadderFilter`).
- **Header-only.** No `.cpp` files in `rpdsp`. Inline everything.
- **API shape:** `prepare(sampleRate, ...)` to initialize sample-rate-dependent
  state, `reset()` to return to a deterministic silent state, `process(...)`
  for per-sample (or sometimes per-block) work. Not `Init()`/`Process()` — if
  porting code from another library (DaisySP-style, Teensy Audio Library,
  etc.), rename to this lowercase convention.
- **Register new headers in `src/rpdsp.h`** (the discovery shim) so
  `#include <rpdsp.h>` continues to pull in every public header; user code
  should otherwise include the specific namespaced header it needs, e.g.
  `#include <rpdsp/oscillator.h>`, not the umbrella file.
- **No allocation, no blocking, no I/O in `process()`.** No `new`/`delete`,
  no STL containers that allocate, no `Serial`, no locks/semaphores/USB/I2C/SPI
  transactions, no unbounded loops. Use fixed-capacity storage
  (`rpdsp::DelayLine<Capacity>`, plain arrays) sized at compile time.
- **Denormals:** call `rpdsp::zapDenormal()` on feedback/long-decay filter
  state (see every filter in `filter.h`, `ladder.h`, `effects.h`).
- **Smoothing:** audible parameter changes go through
  `rpdsp::LinearSmoother` (`setTarget()` off the audio thread or between
  blocks, `next()` per sample inside `process()`) rather than stepping
  directly.
- **Shared helpers** live in `algorithm.h` (`clamp`, `clamp01`, `lerp`,
  `wrap01`, `dbToGain`, `gainToDb`, `midiNoteToHz`, `safeSampleRate`,
  `clampCutoff`, `onePoleSmooth`, `softClip`, `fastTanh`,
  `equalPowerPanLeft/Right`) and `realtime.h` (`zapDenormal`, `XorShift32`).
  Reuse these instead of re-deriving the math in a new module.
- **Attribution:** when porting code from an external MIT/BSD source (as
  `ladder.h` does from the Teensy Audio Library / Infrasonic Audio), keep the
  full original license header at the top of the file plus a short note that
  the rpdsp port follows rpdsp conventions. Mechanical renames only — don't
  change the underlying model/coefficients unless that's the explicit goal.
- **No host test framework exists.** Validate a new module by including it
  from an example (temporarily, if needed) and compiling with `arduino-cli`
  per "Build & validate" above.

## rpdsp module catalog (verified against source, June 2026)

Core utilities — `algorithm.h`, `config.h`, `realtime.h`: math/DSP helper
functions (see above), `kDefaultSampleRate` / `kDefaultBlockSize` /
`RPDSP_BLOCK_SIZE`, `zapDenormal`, `XorShift32` PRNG.

Oscillators — `oscillator.h`: `Phasor` (shared phase base), naive
`SineOscillator`/`TriangleOscillator`/`SawOscillator`/`PulseOscillator`,
band-limited `SecondOrderBSplineSawOscillator`/`SecondOrderBSplinePulseOscillator`/
`SecondOrderBSplineHardSyncSawOscillator`, `NoiseOscillator`.
`hypersaw.h`: `Hypersaw` (multi-voice detuned saw, JP-8000 style).
`hardware_interpolator.h`: `HardwareInterpolatorPool`, `HardwareWavefolder`,
`HardwareOscillator` — wraps the RP2040/RP2350 hardware interpolator
peripheral (`interp_hw_t`/`interp_config`).

Filters — `filter.h`: `OnePoleLowpass`, `DcBlocker`, `BiquadLowpass`,
`StateVariableFilter` (+ `StateVariableOutput`). `ladder.h`: `LadderFilter`
(4-pole Huovilainen/Moog-style, ported from the Teensy Audio Library).

Envelopes & dynamics — `envelope.h`: `ADSR`. `dynamics.h`: `EnvelopeFollower`,
`CompressorStaticCurve`, `CompressorDetector`, `GainReductionSmoother`,
`Compressor` (deliberately split into stages so each is independently
testable).

Effects — `effects.h`: `Waveshaper`, `Delay`, `Chorus`, `CombFilter`,
`AllpassFilter`, `SchroederReverb`, `StereoSchroederReverb`.
`delay_line.h`: `DelayLine<Capacity>` (compile-time-sized circular buffer,
`readLinear()`/`readCubic()` fractional reads). `parameter_smoother.h`:
`LinearSmoother`.

Synthesis — `voice.h`: `TriggeredSynthVoice<MaxOscillators>` (+
`VoiceTrigger`, `SawVoiceOscillator`, `VoiceEnvelopeSettings`,
`VoiceFilterSettings`, `TriggeredSynthVoicePreset`, preset factories).
`waveguide.h`: `KarplusStrongVoice` (plucked-string physical model).

Analysis/metering — `analysis.h`: `ZeroCrossingPitchDetector`,
`YinPitchDetector<WindowSize, MaxTau>`, `RmsPeakMeter`. `clock_tracker.h`:
`ClockTracker` (+ `StepPosition`).

Sequencing/UI helpers — `gate_pattern.h`: `GatePattern`. `rhythm_sequencer.h`:
`RhythmGateSequencer`. `knob_bank.h`: `KnobBank`. `pickup_knob.h`:
`PickupKnob`. `joystick_recorder.h`: `JoystickRecorder` (+ `Position`).

## pico_audio_i2s library

Vendored from Raspberry Pi's `pico-extras` (BSD-3-Clause), packaged as an
Arduino library by IC-Alchemy. Provides the producer/consumer
`audio_buffer_pool_t` API, PIO/DMA I2S transport (`audio_i2s_setup`,
`audio_i2s_connect`, `audio_i2s_set_enabled`), and integer sample-format
conversion helpers. Treat this as a stable dependency boundary — DSP code
should not need to touch its internals; it only needs `audio_new_producer_pool`,
`take_audio_buffer`, `give_audio_buffer`, and the `audio_i2s_*` setup calls.
`extras/audio_i2s.pio` is the human-authored PIO source kept for reference;
the build actually uses the already-assembled `audio_i2s.pio.h` committed in
`src/pico_audio_i2s/` — there is no PIO build step in this repo.

## Docs vs. reality — read this before trusting `Docs/`

`Docs/realtime_rules.md`, `Docs/rp2350_tuning.md`, and
`Docs/algorithm_catalog.md` are largely accurate on *principles* (no
allocation/blocking in the audio path, denormal hygiene, dual-core split,
48 kHz/32-frame baseline, parameter smoothing, measuring worst-case cost) and
those rules should still be followed. But they also describe an aspirational
processing framework and several types that **do not exist anywhere in this
codebase**: `rpdsp::AudioBlock<Capacity>`/`DefaultAudioBlock`,
`rpdsp::MidiByteParser` (+ `MidiNoteEvent`/`MidiControlEvent`),
`rpdsp::GateDebouncer`, `rpdsp::SmoothedAdcParameter`,
`rpdsp::CallbackPerformanceMeter`, `rpdsp::StereoMixer<Channels>` /
`StereoSample`, `rpdsp::I2sSampleBridge`, and a `process(AudioBlock&)`-style
processor interface. `Docs/algorithm_catalog.md` also predates and omits the
five newest headers (`ladder.h`, `waveguide.h`, `clock_tracker.h`,
`analysis.h`, `hardware_interpolator.h`). `Docs/example_Ideas/*.cpp/.md` are
write-ups for sketches that were never built — there is no corresponding code
in `Examples/`.

Practical implication: **don't go looking for these symbols, and don't write
new code against them** — they're not real. The actual processing model is
the per-sample `fill_audio_buffer()` callback shown above. If you need to add
real block-processing or MIDI-parsing support, that's new work, not a wiring
exercise against an existing hidden API. When `Docs/` and the source disagree
on a concrete API, the source under `libraries/rpdsp/src/rpdsp/` wins.

The root `README.md` is close to accurate but has drifted on a couple of
specifics: it lists per-example sample rates of 44.1 kHz for
Oscillators/SimpleOscillators/TwoChannelOscillator, but all three actually
run at 48 kHz in the current `.ino` files, and the README's example list and
repo-layout tree both omit the `Examples/LadderFilter` sketch that already
exists in the repo.

## Git conventions

- Conventional-commit-style subject lines are the norm (`feat(rpdsp): ...`,
  `chore(build): ...`, `fix(...): ...`), but plain descriptive subjects also
  appear in history — match the style of recent commits on the file(s) you're
  touching rather than forcing a format.
- When porting/refactoring across many files in one logical change, prefer
  `git mv` to preserve history, as the rpdsp library-conversion migration did.
- There's no CI gate. Treat a clean `arduino-cli compile` for every affected
  example as the bar for "done."
